## 1. 需求摘要

俱悦酒店小程序 V1.0 是一套面向酒店经营场景的多端系统，覆盖 **用户端微信小程序、保洁端微信小程序、酒店前台工作台、管理端 PC 后台**。系统围绕酒店预订、商城、会员、订单、客房服务、保洁任务、前台入住退房、房价管理、员工权限、内容运营、库存耗材、协议/加盟企业等业务建立线上化闭环。

本 PRD 的定位是将脑图内容整理成产品需求基线。详细功能点已同步输出到 Excel 表：`exports/俱悦酒店小程序V1.0-需求功能表.xlsx`。本文档负责描述产品目标、端口范围、页面结构、核心流程、功能规格、权限边界、非功能需求与待确认问题。

## 3. 项目定位与目标

### 3.1 一句话定位

面向酒店消费者、保洁人员、前台人员与平台管理员的一体化酒店小程序及后台管理系统，实现 **预订交易、入住服务、会员运营、保洁协同、房态管理、商品商城、内容营销与后台管控** 的线上闭环。

### 3.2 建设目标

| 目标编号 | 目标 | 验收指向 |
| --- | --- | --- |
| G-01 | 建立用户端酒店预订闭环 | 用户可搜索酒店、选择入住时间与入住人、查看酒店详情、下单并在订单中跟踪状态 |
| G-02 | 建立会员与资产运营能力 | 用户可查看会员等级、积分、余额、优惠券、签到、分销推广和任务积分 |
| G-03 | 建立入住后服务能力 | 入住用户可查看当前入住酒店/房间，并发起一键开门、灯光控制、续住、客房清洁等服务 |
| G-04 | 建立保洁执行闭环 | 保洁员工可查看今日任务、扫码进入打扫、记录耗材/拍照/报损并形成打扫记录 |
| G-05 | 建立前台运营工作台 | 前台可快速入住、房卡办理、新增订单、查看房态、办理退房和处理用户/订单 |
| G-06 | 建立平台后台治理能力 | 管理员可维护权限、员工、用户、订单、房型、商城、仓储、协议企业、内容与活动 |
| G-07 | 建立数据与报表基础 | 支持首页数据大屏、营业报表、财务报表、库存预警、订单统计等数据查看 |

### 3.3 成功度量

| 指标编号 | 指标 | 建议目标 |
| --- | --- | --- |
| KPI-01 | 用户完成酒店预订流程 | 从首页进入酒店详情并成功提交订单，流程无阻断 |
| KPI-02 | 保洁任务闭环完成率 | 今日任务可创建、领取/进入、完成并在记录中查询 |
| KPI-03 | 前台房态准确性 | 快速入住、退房、打扫中、空闲、已预订等状态在房间管理中同步更新 |
| KPI-04 | 后台配置生效 | Banner、功能区、商品、房型、价格、角色权限等配置后可在对应端展示或限制 |
| KPI-05 | 数据可追溯 | 用户、订单、房型、商品、库存、员工等关键数据具备创建时间、更新时间或操作记录 |

## 6. 核心业务流程

### 6.1 用户登录与入住人完善流程

1. 用户进入小程序，展示启动页。
2. 用户选择手机号登录或微信授权登录。
3. 手机号登录需录入手机号、验证码或密码。
4. 微信授权登录需获取微信信息并绑定微信手机号。
5. 登录前需勾选《用户协议》《隐私政策》《平台协议》，未勾选时给出提示。
6. 首次登录后提示添加入住人信息，包含入住人姓名、身份证号、手机号。
7. 身份证信息需支持绑定，涉及人脸实名认证和人脸二要素认证能力，具体认证服务商待确认。

### 6.2 酒店预订流程

1. 用户在首页输入关键字或选择位置，也可通过热门搜索词、搜索历史进入。
2. 系统获取当前城市，用户可手动选择省/市/区。
3. 用户选择入住时间、离店时间，系统展示共几晚。
4. 用户选择已添加入住人或新增入住人。
5. 点击查询后进入酒店列表。
6. 酒店列表展示封面、名称、位置、服务、价格、评分、活动信息。
7. 用户进入酒店详情，查看介绍、电话、位置、服务、房型、评价、周边配套。
8. 用户选择房型、提交订单并进入支付/待支付状态。
9. 订单状态进入用户端酒店订单列表，同时可被前台和后台查询。

### 6.3 酒店订单处理流程

1. 用户提交订单后，订单进入待支付或待使用状态。
2. 用户端订单列表按全部、待支付、待使用、待评价、已完成、退款中分类展示。
3. 待支付订单支持支付、取消等操作；待使用订单支持查看或后续入住；已完成订单支持评价；退款状态支持查看退款进度。
4. 前台工作台可在订单管理中查看订单详情，并对待入住订单办理入住。
5. 入住成功后房间状态同步为已入住，今日入住列表展示入住人、联系方式、房型、房间号、入住时间。
6. 退房时前台办理退房，订单进入已完成或退房相关状态，同时可触发保洁任务。

### 6.4 入住后服务流程

1. 用户进入服务页，系统展示当前入住酒店、入住房型、房间号、入住时间。
2. 用户可使用智能服务：一键开门、一键开窗、灯光控制、空调控制、电视控制、WIFI 密码。
3. 用户可发起酒店服务：续住、呼叫服务、早餐、客房清洁、洗衣、叫醒、行李寄存、健身中心、叫车。
4. 对于续住服务，用户选择续住时间后提交前台确认。
5. 对于客房清洁，用户端发起后需同步到保洁端形成任务，具体同步规则待确认。

### 6.5 保洁任务执行流程

1. 系统根据今日工作安排、退房数、用户发起的清扫服务或前台创建的打扫任务生成今日任务。
2. 保洁端首页展示今日时间、今日任务、今日打扫间数、已完成、待打扫。
3. 保洁人员通过扫码打扫扫描门牌号。
4. 扫描成功后进入打扫流程，可开始打扫、登记物品更换、拍照或填写打扫完成信息。
5. 物品扫码用于扫描床单等耗材或物品条码/二维码，并展示已扫描物品列表。
6. 打扫完成后生成打扫记录，包含房间号、房型名称、打扫状态、耗材数量、拍照数量。
7. 报损记录需同步至管理端仓储管理的报损记录。

### 6.6 前台入住退房与房态流程

1. 前台工作台展示房间总数、已入住数、空闲数、今日入住、今日退房、今日营收。
2. 前台可通过快速入住办理新客入住登记，填写客户姓名、手机号、证件类型、证件号码、房型、房间号、入住/退房时间、支付方式、押金、备注。
3. 房间管理支持按空闲中、已入住、打扫中、已预订筛选。
4. 空闲房间支持入住办理；已入住房间支持查看详情、退房等操作。
5. 退房后房间状态应进入待打扫或打扫中，并同步保洁端任务。
6. 打扫完成后房间可恢复为空闲状态，具体流转状态需客户确认。

### 6.7 商城与积分兑换流程

1. 用户通过首页金刚区进入酒店商城。
2. 商城首页展示搜索框、购物车、Banner、商品分类、商品列表。
3. 商品列表展示封面、名称、价格、兑换积分、剩余数量。
4. 用户提交商城订单后，订单进入商城订单列表。
5. 商城订单按全部、待支付、待收货、待评价、已完成、退款中分类展示。
6. 管理端维护商品类目、商品上下架、库存、评价、售后和商城订单状态。
7. 积分兑换与库存扣减规则待确认。

### 6.8 内容运营与活动流程

1. 管理端可维护首页 Banner、商城 Banner、首页功能区、周边推荐、平台介绍、活动。
2. Banner 支持图片、标题、内容、绑定链接、有效时间、创建/更新时间。
3. 活动支持发布、添加、删除、修改、刷新，包含活动标题、开始时间、结束时间、活动内容、是否需要报名、报名列表。
4. 用户端首页展示 Banner、活动推荐、特价酒店推荐、周边推荐、商城商品推荐。
5. 用户可在社区动态、协议专区、加盟合作等入口查看对应内容并发起申请。

## 9. 权限与角色控制

### 9.1 权限原则

1. 管理端需支持角色权限分配，不同角色拥有不同菜单和操作权限。
2. 脑图中明确的角色包括管理员、财务人员、前台人员、保洁人员。
3. 权限需覆盖菜单可见、按钮可用、数据范围、关键操作留痕。

### 9.2 建议权限矩阵

| 角色 | 可用端口 | 主要权限 |
| --- | --- | --- |
| 管理员 | 管理端 | 全部模块配置、角色权限、内容、订单、商品、员工、企业、数据 |
| 财务人员 | 管理端 / 前台工作台 | 营业报表、财务报表、订单金额、发票与结算数据 |
| 前台人员 | 酒店前台工作台 | 入住退房、订单处理、房间管理、住客管理、房价查看/部分维护 |
| 保洁人员 | 保洁端 | 查看本人任务、扫码打扫、提交耗材/照片/报损、查看排班和绩效 |
| 普通会员 | 用户端 | 酒店预订、商城、订单、服务、会员、我的 |
| 协议会员 | 用户端 | 普通会员能力 + 协议专区与协议价格相关能力 |

| 人脸认证服务 | 首次入住人提醒 | 身份证绑定、人脸二要素认证 | 服务商、费用、失败处理 |

## 11. 非功能需求

### 11.1 易用性

- 小程序核心路径（搜索酒店、选择时间、选择入住人、查看房型、下单）需保证步骤清晰。
- 前台工作台应突出房态、今日入住、今日退房和快捷操作。
- 保洁端应适配移动场景，扫码、开始打扫、完成提交路径尽量短。

### 11.2 稳定性

- 订单、支付、房态、保洁任务、库存等关键链路需保证状态一致。
- 支付和退款需支持幂等处理，避免重复支付或重复退款。
- 前台退房和保洁任务同步失败时需有重试或人工补偿机制。

### 11.3 安全性

- 用户隐私信息如手机号、证件号需脱敏展示，脑图中前台订单已明确联系方式脱敏处理。
- 管理端和前台工作台需账号密码登录。
- 关键操作如删除、冻结、解封、退款、价格修改、权限变更需记录操作日志。

### 11.4 可配置性

- Banner、功能区、周边推荐、活动、商品、类目、房型、价格、权限角色、短信模板等应后台可配置。
- 会员等级、权益、积分、优惠券、分销佣金、签到和任务规则建议配置化，具体规则待确认。

### 11.5 数据留痕

- 用户、员工、商品、企业、订单等列表脑图中包含创建时间、更新时间时，应在数据库与后台展示中保留。
- 管理员日志、订单操作、房态变更、保洁完成、库存出入库、报损记录应可追溯。

## 14. 交付与开发流程

脑图中的开发流程包含：

1. 产品需求：需求洽谈、需求分析、核心业务需求确认、需求整理、技术分析、内部立项，并提供技术方案和技术架构咨询。
2. 产品业务设计：思维导图、功能逻辑图、系统架构图。
3. 产品设计：UE 原型设计、UI 视觉设计、原型策划、交互设计、效果设计、高保真制作、UE 确认、UI 验收。
4. 产品研发：服务器搭建、数据库设计、产品架构设计、后台管理架构设计、前后端代码编写。
5. 产品测试：功能测试、性能测试，并输出测试报告。
6. 产品交付：上线部署与交付。
7. 产品售后：在线支持、远程协助、信息反馈、7x24 小时响应、数据库备份提醒、常见问题解答、定制化功能开发。

开发周期脑图中记录为：**UI 确定后 XX 工作日**，具体周期需在需求冻结后结合范围评估。
