Part 01

产品概述

中医网平台是面向中医健康领域的多商户电商与服务平台，为消费者提供中医商品、健康设备和到店服务的一站式购买入口；为商户提供低门槛入驻、店铺自主经营、营销推广与销售分润的完整经营工具；为平台运营方提供商户审核、订单监管、批量结算、营销配置等运营管控能力。一期完成"自营 + 商户入驻"双轨上线，支撑 4000+ 商户体量，奠定后续直播、问诊、处方流转等扩展业态的能力底座。

Part 02

需求分析

维度说明
需求背景客户原有"金翼商城""金翼防辐射小程序"两款产品在中医健康类目运营效果不佳，定位单一、缺少商户生态与活动能力。新建中医网平台聚焦中医健康赛道，打通"商品 + 设备 + 服务"三类业态，引入多商户入驻机制，已沉淀 4000+ 意向商户。
核心诉求① 平台运营方：批量审核入驻、批量打款结算、营销活动可视化配置、平台费率与提现门槛可后台动态调整。② 入驻商户：自助入驻并支付年费、店铺与商品/服务自主管理、订单及时处理、收益清晰可视、提现快速到账。③ 终端消费者：一站式选购可信中医商品与服务、参与营销活动获取优惠、阅读中医知识内容。
目标用户画像平台运营方（平台管理员）、入驻商户（中医馆 / 养生机构 / 艾灸刮痧服务商 / 中医产品品牌方）、终端消费者（关注中医养生与健康消费的中青年群体，覆盖一二三线城市）。
交付范围（一期）消费者端（uni-app：APP + 微信小程序）、商户端（PC + 移动 H5 双形态）、平台管理端（PC Web）三端全链路；含 UI 全套设计、前后端开发、微信支付与企业打款对接、上线部署。
关键假设① 微信支付商户号、企业付款资质、平台 ICP 与备案由甲方提供。② 商户入驻仅需营业执照 + 法人身份证 + 联系信息，不涉及医疗机构执业许可（一期）。③ 内容文章以图文为主，不含视频上传与直播推流（二期）。④ 服务预约时段以"日 + 时段"颗粒度，不做精确分钟段。

类别要求

合规商户营业执照真实性核验（OCR + 工商接口）；中医健康类目内容人工 + 关键词双重审核；遵循《个人信息保护法》《电子商务法》及微信小程序类目运营规范。

Part 04

技术选型

按"后端 PHP → Java → Python → Go"与"前端 Vue → uni-app → HTML → 原生"的优先级规则匹配本项目特性，结合 4000+ 商户体量与跨端需求作出选型。

层级选型理由
后端PHP电商平台对快速迭代和成熟生态依赖度高，PHP 在多商户电商场景成熟方案多、人才储备充足、单价合理，能满足 4000 商户体量下的查询与交易并发需求。
前端（消费者端）uni-app一套代码同时输出 iOS APP、安卓 APP 与微信小程序，相比原生方案节省约 40%~50% 前端工时；针对秒杀、商品详情等性能敏感页面采用 nvue 局部优化。
前端（商户端）Vue（H5）商户端以表单与列表为主，H5 形态便于商户在 PC 与手机两端使用，无需独立 APP 包，迭代上线无需审核。
管理后台（PC Web）Vue组件库成熟、表单与表格组件丰富，适合后台管理类场景；可与商户端共用基础组件库降低维护成本。
UI 设计全套视觉稿含三端全套页面视觉设计、设计规范、切图与图标资源，覆盖中医健康行业调性的色彩与排版方案。

Part 05

技术难点与解决方案

技术难点解决方案要点
4000+ 商户的店铺数据隔离与权限控制采用"商户 ID 强隔离 + RBAC 角色权限"双层控制：所有店铺业务表带 merchant_id 索引，接口层强制注入当前商户 ID 过滤；后台角色权限按"平台管理员 / 运营 / 财务 / 商户主账号 / 子账号"分级；越权访问双重校验（用户态 + 资源归属）。
秒杀活动的高并发与防超卖秒杀库存独立计数（与普通库存隔离）；Redis 原子扣减 + Lua 脚本保证库存一致性；下单走异步队列削峰；前端按钮防重复点击 + 后端订单幂等校验；活动开始前预热数据到 Redis 避免缓存击穿。
分销订单佣金计算与多角色账务一致性分销关系绑定后写入 promoter_id 字段；订单完成后由账务服务统一结算（消费者付款 → 平台分润 → 商户应收 → 推广佣金）；账务流水按订单维度全留存，支持按商户/月份对账；佣金 T+N 释放，防止退款撤销。
商户批量打款的资金安全与失败重试提现单生成时锁定可提现余额；批量打款走"预生成打款单 → 调用企业付款 API → 异步回调更新状态 → 失败重试 + 人工介入"四步闭环；所有打款记录全留存（操作人 / 时间 / IP）；失败原因（余额不足 / 卡号错误）回写并通知。
服务预约的时段冲突与扫码核销每个商户的可预约时段配置为"日 + 时段 + 名额"三元组，预约下单前实时校验剩余名额；核销码采用一次性签名 token（含订单 ID + 过期时间）；商户扫码后服务端校验签名 + 状态 + 归属，核销成功后状态机推进至"已完成"。
多商户订单拆单与结算对账购物车按商户分组结算，下单时若涉及多商户自动拆单（同一支付单 → N 个子订单）；每个子订单独立履约与结算；月度对账报表按商户聚合（GMV / 平台抽佣 / 应付商户）。
中医健康类目内容合规审核商户上架商品 / 服务标题与描述过关键词词库（涉医、虚假宣传词）；文章发布走"自动初审 + 人工复审"双层；图片调用腾讯云天御 / 阿里云内容安全识别；命中违规自动驳回并通知。
跨端（APP / 小程序 / H5 / PC）数据一致性统一 RESTful 接口规范；登录态采用 Token + Refresh Token，跨端共享；订单 / 优惠券等核心数据以"以服务端为准"原则同步，端侧仅做展示缓存；版本变更通过接口版本号控制兼容。
Part 06

技术实施方案

系统采用前后端分离架构。表现层一套 uni-app 代码同时输出 iOS / 安卓 APP 与微信小程序，商户端与平台后台基于 Vue 构建 H5 与 PC Web；服务层基于 PHP 构建 RESTful API 网关，按业务域拆分商品、订单、支付、商户、营销、分销、内容等核心服务模块；数据层 MySQL 主存储 + Redis 缓存 + MQ 异步削峰。

表现层
uni-app（iOS APP） uni-app（安卓 APP） uni-app（微信小程序） Vue（商户 PC + H5） Vue（平台 PC 后台）
服务层

MySQL 主从（核心业务） Redis（缓存 / 秒杀库存 / Token） 消息队列（订单异步 / 通知） 腾讯云 COS（图片 / 视频 / 资质） 日志与审计存储
交付层

方案边界说明

  • 本方案以"前后端分离 + 单体服务模块化"形态实施，不引入微服务治理框架，避免一期过度设计；二期业务体量进一步增长后可平滑拆分为微服务架构。
  • 服务器部署形态默认为单机房双节点（应用 + 数据库），含负载均衡入口；甲方如有阿里云 / 腾讯云资源池要求，按最终选定厂商部署。
  • 本方案不含直播推流、IM 在线问诊、电子处方流转、医保对接等扩展能力（属二期范围）。
  • 第三方平台（微信、支付宝、短信、OCR 等）的接入资质与账号由甲方提供，乙方负责接口对接开发。
