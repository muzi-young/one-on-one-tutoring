社区洗车小程序

多门店 · 柜体存取钥匙 · 监控查看
本项目面向社区地下车库场景，为业主提供线上预约洗车、柜体自助存取车钥匙、服务过程监控查看的完整闭环解决方案。系统覆盖用户端小程序、柜体平板控制面板、PC后台管理系统，并对接千鸣云柜体总控与监控设备，支持多门店运营与多角色权限管理。

建设目标
  1. 实现用户从选门店、选服务、填信息、支付到派单的完整线上预约闭环，减少人工沟通成本。
  2. 通过柜体平板与千鸣云API对接，实现钥匙存取全程自动化，每次开门动作可追溯。
  3. 建立强权限监控查看机制：用户仅能在已支付且服务中查看对应门店监控，保障隐私安全。
  4. 支持多门店运营与总部-店长-员工多角色权限体系，为后续规模化扩展奠定管理基础。

PART 05
技术难点与方案
本项目需要重点攻克的技术问题

1. 千鸣云柜体API限频与并发控制

  • 厂商限制：单 sign 3次/分钟，单设备 100次/分钟，单 appKey 5000次/分钟。
  • 方案：后端实现请求队列与合并机制，对同一设备的连续指令做节流；引入分布式锁防止并发分箱冲突。

2. 设备回调幂等性保障

  • 厂商回调（开门记录/测物结果）可能重复推送，需保证订单状态与操作日志不重复写入。
  • 方案：以 Base62(deviceId+openDate+boxNo+offlineRecordId) 为幂等键，入库前做唯一性校验。

3. 监控查看强权限控制

  • 用户仅在「已支付 + 服务中」时可查看监控，完成后立即失效；管理员可随时查看。
  • 方案：播放鉴权接口实时校验订单状态，token 设置短有效期（5分钟），播放页定期心跳校验权限。

4. 柜体离线与服务降级

  • 柜体断网或厂商服务异常时，需阻止用户进入「存钥匙」阶段，避免订单卡死。
  • 方案：下单前调用 chkDevice 校验在线状态；开门指令超时未回执触发 query_boxInfo 补偿查询。

5. 多门店数据隔离与权限链

  • 店长只能查看本店数据，员工只能操作本店订单，总部可查看全部。
  • 方案：所有业务表增加 store_id 字段，接口层统一挂载当前用户所属门店范围进行数据过滤。
PART 06
技术实施方案
开发、测试与交付的标准流程

开发流程

  • 采用前后端分离架构，接口优先：先定义 API 文档（Swagger），再并行开发。
  • Git 分支管理：main / develop / feature / hotfix，所有代码变更通过 Pull Request 审查合并。
  • 接口联调：本地环境 → 测试环境 → 预发布环境 → 生产环境，四层隔离。

测试策略

  • 单元测试：后端核心服务（订单状态机、柜体指令签名、权限校验）覆盖率不低于 70%。
  • 集成测试：模拟千鸣云回调，验证开门成功/失败/超时各分支的状态回写正确性。
  • 验收测试：甲方业务方参与，覆盖完整下单→存钥匙→取钥匙→还钥匙→取回钥匙主流程。

交付标准

  • 源代码完整交付（Git 仓库权限移交），含部署文档与接口文档。
  • 提供 3 个月免费维护期：Bug 修复、紧急问题响应（工作日 4 小时内）。
