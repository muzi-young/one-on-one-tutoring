# 洗车小程序（多门店+柜体存取钥匙+监控查看）需求文档 PRD

版本：V1.0  
目标：用户端小程序、柜体端面板、后台管理、监控接入与权限、硬件对接与数据沉淀、多门店多角色

---

## 1. 项目背景与核心目标

### 1.1 背景

- 场景：社区地下车库（负一楼为主）提供**有人洗车**服务。
- 用户（业主）在回家后于单元楼门口/负一楼入口处，通过**柜体控制面板**或**扫码二维码**进入小程序完成下单。
- 员工按订单信息上门取钥匙/送回钥匙，并执行洗车服务。

### 1.2 核心目标（一期 P0）

- **预约下单闭环**：选时间 → 选服务方式（上门取送/自到店）→ 填车辆与车位信息 → 拍照留证 → 支付 → 派单/处理 → 完成。
- **柜体存取钥匙闭环**：订单驱动柜门开合（用户/员工/归还各环节），后台可追踪柜体与格口状态。
- **监控查看（强权限）**：
  - **项目方/管理员**：可随时查看门店监控；
  - **下单用户**：仅在“订单已支付且服务进行中”可查看对应门店监控；
  - **未下单**、**订单已完成**均不可查看。
- **多门店**：小程序可按定位/搜索选择门店；后台支持总部/门店多角色管理。

## 3. 业务流程（P0）

### 3.1 用户下单（扫码/面板两入口同流程）

1. 用户在柜体处**扫码二维码**进入小程序（或在柜体平板上直接进入固定入口）
2. 选择门店（默认最近/扫码门店锁定）
3. 选择服务项目
4. 进入“服务下单”四步表单：
  - Step1 选择服务时间（日期+时间段）
  - Step2 选择服务方式（是否需要上门取送车/钥匙）
  - Step3 填写车辆信息（车牌、车位地址：楼栋/单元/楼层/区域/车位号）
  - Step4 拍照留言（车辆照片/车位照片/备注）
5. 勾选服务协议 → 确认下单 → 线上支付（可选线下）
6. 支付成功生成订单，进入“待处理”

### 3.2 派单与服务执行（P0 简化）

- 后台/店长查看“待处理订单” → 指派员工（或默认门店队列）  
- 员工到柜体取钥匙（触发柜门开合）→ 上门取车/执行洗车 → 归还钥匙（触发柜门开合）  
- 订单完成

### 3.3 监控查看权限（P0 强约束）

- 管理员：随时可看本门店（或全门店）实时监控
- 用户：
  - 必须满足：**订单已支付 + 订单状态=服务中**（或“已接单/进行中”）
  - 不满足：提示“仅限服务进行中可查看”
  - 订单完成后：入口置灰/隐藏（不可回放）

### 4.2 柜体端（平板控制面板 + 柜体总控）需求

#### 4.2.1 目标

- 在单元楼负1楼门口投放**存放钥匙柜体**，中间安装**平板控制面板**。
- 平板用于“下单入口/扫码入口/柜门开合操作入口”，且需**锁死**，防止跳出做其他用途（属于部署/运维要求）。

#### 4.2.2 柜体硬件组成（客户自购）

- 柜体（定制或网购）+ 格口（可多格）
- 平板（安卓平板，批量采购）
- 柜体“总控模块”（控制继电器/电磁锁等开关）
- 网络：优先 WiFi（客户表示可牵网），可扩展 4G

#### 4.2.3 小程序是否能“控制总控模块”（回答客户问题）

**结论**：可以，但取决于总控模块/厂商能否提供对接协议与鉴权能力。  
建议对接链路（P0 推荐）：

1. 小程序/柜体平板上的操作 → 调用后端接口（HTTPS）
2. 后端通过 IoT 通道向“总控模块”下发开门/关门指令（MQTT/WebSocket/HTTP 回调均可）
3. 总控模块执行开锁并回传结果（成功/失败/超时）

**不建议**（风险高/受限）：

- 直接在小程序里通过蓝牙/局域网直连控制（兼容性与安全、运维复杂度高）

#### 4.2.4 柜门开合业务动作（P0，已对照硬件接口）

> 已确认硬件接口为 **千鸣云 WebApi-1.4.1**，详见第 7 章接口对接方案。下表给出每个业务动作对应的厂商接口。


| 编号  | 动作             | 触发方         | 厂商接口                                          | 关键参数                             |
| --- | -------------- | ----------- | --------------------------------------------- | -------------------------------- |
| A   | 用户存钥匙（下单后）     | 用户（柜体面板/扫码） | `getBox`（自动分箱） + `openBoxCQ`                  | `optType=1 存`                    |
| B   | 员工取钥匙（接单后）     | 员工（员工端/后台）  | `openBoxCQ` 或 `openOneBox`                    | `optType=2 取` 或 `optType=3 中途开箱` |
| C   | 员工还钥匙（服务完成前/后） | 员工          | `openBoxCQ`                                   | `optType=1 存`                    |
| D   | 用户取钥匙（完成后可取）   | 用户          | `openBoxCQ` + `getBox`（释放）                    | `optType=2 取`                    |
| E   | 管理员强开（异常处理）    | 后台          | `openOneBox` / `openAllBox` / `openMulitBox`  | —                                |
| F   | 状态巡检           | 后端定时任务      | `chkDevice` / `query_boxInfo` / `get_boxInfo` | —                                |


每次动作必须落库的字段：

- 订单号、门店、柜体ID（cabinetId）、格口号（boxNo）
- 操作人类型（用户/员工/管理员）、操作人ID
- 操作时间、操作结果（厂商 code：200成功/250失败/...）
- 关联的设备主动回调记录ID（offlineRecordId）
- 设备测物结果（status：0关 / 1开；箱内有物 0/1/2）

## 5. 关键业务规则（必须在评估前确认）

### 5.1 下单与柜体动作关系

需要确认的策略（建议默认）：

- **支付成功后**才允许触发“用户存钥匙/开格口”（防止占用格口）
- 若允许“先存后付”，需增加超时释放与黑名单策略（P1）

### 5.2 监控查看权限（强规则）

- 用户仅限：已支付 + 服务中  
- 完成后不可看（按客户明确要求）
- 未下单不可看

### 5.3 线下支付（P1/可选）

- 开启后：用户下单选择“线下支付”，由门店收银在后台手动改为已支付
- 需要风控：限制线下支付的使用场景（例如仅门店到店服务）

## 7. 第三方/硬件对接范围（P0）

### 7.1 柜体总控对接（千鸣云 WebApi-1.4.1，已确认）

> 文档版本：2022.03.12 V1.4.1（来自客户提供：WebApi-1.4.1.pdf）

#### 7.1.1 平台与基础信息

- **管理后台**：`https://open.qianmingyun.com/`（新增网点 → 新增设备 → 获取 appKey 与 secretKey）
- **API 域名**：`https://open.qianmingyun.com/webapi/apiService/v1/s1/{接口名}`
- **设备 socket**（厂商内部）：`47.110.67.166:8282`（我方不直接连接）
- **请求方式**：HTTP POST，`Content-Type: application/json`，编码 UTF-8
- **设备编号**：`cabinetId`（屏幕左下角纯数字 / 设备二维码值；前导 0 不传）

#### 7.1.2 鉴权与签名

- 所需密钥：`appKey + secretKey`（在后台「查看 appKey」获取）
- 签名规则：
  1. 必填四参 `cabinetId / appKey / timeStamp(毫秒) / sign`
  2. 所有参数按 ASCII 升序排序，按 `key1=value1&key2=value2` 拼接
  3. 末尾追加 secretKey
  4. 整串做 `MD5（32 位小写）` → 得到 `sign`
- 含中文参数需先做 UTF-8 UrlEncode

#### 7.1.3 调用频次限制（必须在我方后端做合并/节流）


| 维度            | 限制        |
| ------------- | --------- |
| 单 sign        | 3 次/分钟    |
| 单设备 cabinetId | 100 次/分钟  |
| 单 appKey      | 5000 次/分钟 |


#### 7.1.4 关键接口（按本项目场景精选）


| 接口          | URL（接口名）                                        | 用途                          | 业务对应    |
| ----------- | ----------------------------------------------- | --------------------------- | ------- |
| 查询设备在线      | `chkDevice`                                     | 监控柜体在线状态                    | 后台巡检    |

| 设置文本/广告     | `set_text` / `setRotationChart`                 | 柜体屏文案/海报                    | 运营      |
| 设置箱门总数      | `set_totalBoxNum`                               | 配置柜体格口数                     | 部署初始化   |
| 设置工作模式      | `setWorkMode`                                   | 1.刷卡 3.6位密码 4.8位密码 5.手机号+密码 | 安全策略    |

| 强开全部        | `openAllBox` / `clearAllBox`                    | 维护场景                        | E       |
| 设置开箱密码      | `setOpenCode6` / `setOpenCode8` / `setOpenCode` | 给用户/员工临时密码                  | 备用方案    |
| 初始化箱门       | `initBoxs`                                      | 重置数据（慎用）                    | 部署/运维   |


> **本项目推荐主链路**：用「**自动分箱 + 无密码开箱（带存取标志）**」即 `getBox` + `openBoxCQ`，避免管理密码与数据冲突。

#### 7.1.5 必须实现的「设备主动回调接收」接口（我方后端提供 URL）

> 必须由我方提供 URL 给厂商后台配置，且收到后**返回小写 `success` 字符串**，并保证幂等。


| 回调          | 用途                    | 我方处理                            |
| ----------- | --------------------- | ------------------------------- |
| 5.1 扫码回调    | 用户在柜体扫码               | 记录“用户从哪台柜体进入”，关联订单上下文           |
| 5.2 开门记录回调  | 开门成功/失败、存/取/中途取物、开箱方式 | **回写订单状态与柜体操作日志（核心）**           |
| 5.3 测物结果回调  | 箱门开关检测 + 箱内有物监测       | **判断“客户是否真的放进了钥匙 / 员工是否真的取走了”** |
| 5.5 CA 特殊回调 | 设备端密码丢失补救             | 自动重设密码或人工处理                     |


幂等键建议：`Base62.encode(deviceId + openDate + boxNo + offlineRecordId)`

#### 7.1.6 错误码


| code | 含义                     |
| ---- | ---------------------- |
| 200  | 成功                     |
| 250  | 失败                     |
| 202  | 数据有误/参数格式不对/关键数据缺失     |
| 203  | 数据有误                   |
| 206  | 签名校验失败                 |
| 207  | 同一条信息调用次数受限（单 sign 限频） |
| 208  | 柜组调用次数受限               |
| 210  | appKey 下未找到该设备         |


#### 7.1.7 异常与兜底设计（我方必做）

- **网络/限频重试**：207/208 触发指数退避；250/203 上报到后台告警
- **超时未回执**：触发查询接口 `query_boxInfo` 校验真实状态
- **离线**：`chkDevice` 提示设备离线，订单不允许进入“存钥匙”阶段
- **格口冲突**：`getBox` 并发由我方加分布式锁，避免多人分到同一格口
- **测物异常**：5.3 回调若“开门正常但箱内无物” → 提示用户/员工补放/补取

## 8. 非功能性需求

- **安全**：监控与开门指令必须鉴权；所有敏感接口做签名/权限校验；操作日志可追溯
- **稳定性**：柜体开门指令超时重试、离线提示；设备在线心跳
- **性能**：订单列表与监控鉴权接口需在可接受时间内响应（<2s 目标）
- **兼容**：微信小程序；后台 PC 浏览器优先 Chrome
- **隐私合规**：告知监控用途与范围，避免拍摄公共区域；用户协议与隐私条款可配置
