## 2. 背景与目标

### 2.1 需求背景

俱乐部主营马术教培，具备线下门店与课程体系；希望通过线上化承载 **课程售卖、装备售卖、活动竞赛传播、约课与课时管理**，并与门店核销、教练执行闭环打通。

### 2.2 业务目标

| 目标维度 | 描述 |
|---|---|
| 交易线上化 | 课程与装备支持浏览、下单、支付（法规允许范围内），到店核销履约 |
| 运营触达 | 竞赛获奖、开业/充值等活动可向客户批量推送 |
| 教务效率 | 家长自助约课；教练执行点名/扣课；系统自动推送课时与课前提醒 |

### 2.3 目标用户

| 用户类型 | 特征与诉求 |
|---|---|
| 家长 / C 端学员 | 选购课程与装备、导航到店、核销；购买课包后续约课；接收课时与课前提醒 |
| 教练 | 查看当日课程安排；记录上课与扣课（对标访谈中「小麦」类销课体验） |
| 门店前台 / 教务（隐含） | 到店核销；必要时协助改约、补课（是否本期需确认） |
| 运营人员（隐含） | 配置活动推送、竞赛资讯、课程与装备上架 |

### 2.4 核心价值主张

「选课购课 → 到店核销 → 课包扣减与提醒一体化」，辅以装备商城与赛事活动传播。

### 2.5 成功指标（建议，待与客户对齐）

| 指标 | 说明 |
|---|---|
| 线上订单占比 | 课程/装备订单来自小程序的比例 |
| 核销成功率 | 下单订单在规定时间内完成核销的比例 |
| 约课自助率 | 家长自助完成约课占比 |
| 消息到达与点击率 | 课前提醒、课时推送、活动推送的到达率 |

## 5. 业务流程

### 5.1 主流程：单次课程购买与核销

```text
浏览分类 → 课程详情 → 下单支付 → 订单详情（地址/导航/核销凭证）
         → 到店出示核销码 → 门店核销成功 → （若为单次卡）履约结束；
            若为课包则在履约或首次核销时写入课包（规则待定）
```

### 5.2 主流程：课包购买 → 约课 → 上课扣课 → 提醒

```text
购买课包（①线上下单支付生成课包 / ②线下签约由管理员在后台录入课包并绑定学员）
  → 家长在小程序选择时段、教练、课型发起预约
  → 系统校验（教练档期、余位、课包余额等）通过后即时自动确认，无需人工审核
  → 课前一日推送提醒
  → 到店上课：**前台/核销员先完成到店核销**（核销与本节预约或履约凭证绑定）
  → **教练端在「已核销」前提下点名扣课**（扣课流水强制关联本节预约单）
  → 系统推送「本次扣减 1 节，剩余 N 节」
```

**规则摘要（课包）**：**无预约则不扣课**（默认路径）；**未核销则不能扣课**。单次购买体验课等「非课包」订单仅走核销结案，不产生扣课流水。

### 5.3 分支与异常（需求层）

| 场景 | 触发条件 | 系统处理（原则） | 用户反馈 |
|---|---|---|---|
| 支付超时 | 未在规定时间内支付 | 订单关闭或待支付超时 | 提示重新下单 |
| 退款 | 用户申请或规则触发 | 按退款规则退回；已核销订单限制退款（规则待定） | 退款进度可查 |
| 改约 / 缺席 | 家长改时间或未到 | 是否允许自助改约、是否扣课时（商务规则） | 提示规则与结果 |
| 教练临时不可用 | 教练请假 | 后台调整或可触发通知家长改约（是否本期） | 推送通知 |
| 重复核销 | 同一凭证二次核销 | 拒绝并记录日志 | 前台提示失败原因 |
| 未预约扣课（课包） | 教练对学员发起扣课但该节课无有效预约 | **拦截扣课**；可提示先完成预约 | 教练端错误提示 |
| 未核销扣课 | 本节预约未完成到店核销 | **拦截扣课** | 教练端提示先核销 |
| 随到随堂（例外） | 家长临时到店未预约 | 默认不走扣课；是否允许 **管理员后台补预约/补核销** | **P2**：后台补救流程（可选），首期可不做 |

## 11. 详细分析（访谈 → 方案的映射）

### 11.1 已明确事实（来自访谈）

- 课程线上售卖 + 详情浏览；分类为 **体验课、考级课、正式课、大师课**（共四类）。
- 下单后需提供 **地址并可导航**；存在 **门店核销**。
- **竞赛获奖**与 **开业/充值活动** 需要向客户 **推送**。
- **装备**（如头盔）可 **线上下单**。
- **约课**：自选时间、教练、课程类型；客户口语「自动约课」。
- **课包**：举例 100 节；每次上课后 **推送剩余课时**。
- **课前一日提醒** 上课时间。
- **教练端**：类似小麦，执行 **点名/扣课**。

### 11.2 合理推断（部分已升格为定稿）

| 推断 | 理由 | v1.1 状态 |
|---|---|---|
| 需后台管理课程、装备、教练档期、推送内容 | 否则无法运营闭环 | 仍为基线假设 |
| 课包可能与「课程订单」或单独 SKU 绑定 | 访谈未细分 SKU 形态 | 仍适用；且已明确 **线下课包走后台录入** |
| 「自动约课」= 系统校验通过后即时确认 | 原待确认 | **已定稿** |

### 11.3 需求方书面确认纪要（v1.1）

| 项 | 定稿内容 |
|---|---|
| 端侧 | **只做小程序**（不做 App 首期） |
| 约课 | **系统自动确认**（无人工审核环节） |
| 线下课包 | **管理后台录入**，**管理员**操作 |
| 门店 | **目前单店**（方案按单店实现，多店为后续迭代） |

### 11.4 需求方书面确认纪要（v1.2）

| 项 | 定稿内容 |
|---|---|
| 扣课与预约 | **课包扣课必须以有效预约为前提**（默认不支持「随到随扣」）；临时到店补救可走 **P2 后台补录**（可选） |
| 核销与扣课顺序 | **先核销，后扣课**（教练端在未核销状态下不可扣课） |
| 竞赛传播 | 竞赛结果 **允许分享给其他微信用户或其他家长**（小程序转发）；订阅推送受众仍可后台配置 |

### 11.5 与客户初次口径的差异处理

访谈前半段一度归纳为「只有线上课程下单」，后半段补充商城、推送、约课、扣课等；本文以 **信息更完整的后半段** 为主口径写入范围，并在待确认问题中请客户终审优先级。
