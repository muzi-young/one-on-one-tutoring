## 海外（乌兹别克斯坦）教育机构「线上+线下结合」教育APP

## 1. 需求背景与核心目标


| 项目项  | 说明                                                   |
| ---- | ---------------------------------------------------- |
| 业务背景 | 客户在**乌兹别克斯坦**有海外教育业务，线下已运营，准备发展线上业务，需要一套“数字化系统解决方案”。 |
| 机构规模 | **10+门店/分支**（类似国内连锁教育机构）。                            |
| 教育类型 | **语言培训**、**职业技能**、包含部分 **K12**。                      |
| 教学形态 | **线下为主 + 线上拓展**（线上线下互相导流/互补）。                        |
| 交付诉求 | **独立部署**、可定制开发/迭代，不走SaaS。                            |


---

## 2. 关键角色与端侧（建议）


| 角色/端           | 主要诉求                                             |
| -------------- | ------------------------------------------------ |
| 学员端（App）       | 选课/预约/上课（录播/直播）/支付或线下缴费、查看课程权益、消息通知、咨询（AI问答/客服）。 |
| 家长端（可选）        | K12相关：购买/续费/课表/通知（是否需要待确认）。                      |
| 老师端（App/网页）    | 课程安排、授课（直播）、学员管理（深度教学能力待确认）。                     |
| 教务端/排课端（后台/PC） | 自动排课、人工调整、班级/一对一安排、老师资源管理。                       |
| 门店/校区管理员（后台）   | 门店运营、订单与财务、学员与课程管理、权限管理。                         |
| 总部/集团管理员（后台）   | 全局数据、门店管理、统一规则、跨门店权限与数据查看。                       |
| 销售/顾问（后台）      | 续费提醒、跟进、业绩归因、客户通知。                               |


---

## 3. 核心业务流程（现阶段可落地）

### 3.1 线上线下结合（支付与开课权益）


| 流程场景          | 期望流程                                                   |
| ------------- | ------------------------------------------------------ |
| 线上直接购买        | 学员在App购买课程/套餐 → 在线支付（信用卡/本地三方）→ 开通权益 → 预约/上课。          |
| 线下已付款（开通线上权益） | 线下现金/刷卡购买课程 → 后台给学员账号开通套餐/权益 → 学员登录App学习与预约（无需再次支付）。   |
| 线上下单+线下支付（核销） | App选择课程/预约 → 生成订单（未支付/待到店）→ 门店线下收款 → 后台/门店核销确认 → 开通权益。 |


### 3.2 排课与授课（小班/一对一为主）


| 流程场景 | 期望流程                                           |
| ---- | ---------------------------------------------- |
| 自动排课 | 教务配置规则（老师空闲、课程类型、时段等）→ 系统自动分配 → 教务可手动调整。       |
| 学员预约 | 学员选择课程/老师/时间（或仅选时间由系统分配）→ 生成预约 → 通知老师/学员 → 上课。 |

## 4. 需求范围（功能点清单，按模块拆分）

> 说明：当前纪要信息不完整，以下为“从对话明确提到的需求 + 为闭环补齐的必要最小功能”。未提及但推断补齐的项，已用 **【隐性补齐】** 标识。

### 4.1 多语言与国际化（核心）


| 功能点                   | 端侧     | 优先级 | 业务规则/备注                              |
| --------------------- | ------ | --- | ------------------------------------ |
| 系统语言切换：中文/英文/俄语/乌兹别克语 | App/后台 | P0  | 明确提出“四语言切换”。                         |
| 多语言内容维护方式A：语言包/自动翻译   | App/后台 | P1  | 对话提到“语言包自动翻译，可能额外收费”。适合快速上线，但翻译质量风险。 |
| 多语言内容维护方式B：后台人工维护翻译   | 后台     | P0  | 对话提到“后台上传/配置翻译更准”。建议作为主方案。           |
| 【隐性补齐】多语言字段覆盖校验       | 后台     | P0  | 上架课程/套餐时，缺少某语言则阻止发布或提示。              |


### 4.2 门店/组织架构与权限（连锁模式）


| 功能点                      | 端侧     | 优先级 | 业务规则/备注                   |

| 总部-门店层级权限                | 后台     | P0  | 总部可看全局，门店只看本门店；支持跨门店授权查看。 |
| 账号与角色权限（教务/销售/门店管理员/总部等） | 后台     | P0  | 对话强调“设置权限、账号”。            |
| 【隐性补齐】门店地理位置/地址管理        | 后台/App | P1  | 对话末尾提到“地理位置可能需要考虑”。       |


### 4.3 课程体系（录播/直播/线下课、套餐）

## 5. 非功能性需求与约束


| 类别      | 需求/约束            | 备注                          |
| ------- | ---------------- | --------------------------- |
| 部署模式    | **独立部署**（可定制/迭代） | 明确不走SaaS。                   |
| 国际化     | 多语言内容可维护、可校验     | P0要求。                       |
| 支付合规    | 信用卡与本地三方支付需合规接入  | 建议优先适配国内合规供应商的能力/经验，再映射到当地。 |
| 权限与数据隔离 | 总部/门店权限隔离        | 连锁必需。                       |
| 可扩展性    | 支持分期迭代：直播、AI深化等  | 对话提到可做二期三期。                 |

## 7. 待确认问题（关键，不确认会影响报价/工期）

1. **端侧形态（已确认）**：学员端为移动App；老师/教务/管理/销售统一为后台PC。
2. **账号体系**：学员是否需要家长账号？K12占比多大？家长端是否必须？
3. **直播课范围（已确认）**：一期上线，最小范围为“进入直播 + 老师开课 + 互动 + 白板展示 + 连麦 + 课堂测验”。
4. **预约规则（部分已确认）**：取消/改期规则后台可配置；默认策略为：**提前48小时可改/可取消、每单最多改3次、默认不扣课时**；其余差异（最少开班人数、缺勤处理等）待确认。
5. **套餐计费模型（已确认）**：混合（课时+有效期）；是否支持“冻结/延期”待确认。
6. **财务口径**：应收/应付具体指向（老师结算？门店分账？总部抽成？）与对账报表维度。
7. **销售业绩规则**：业绩归因（下单人/跟进人/门店）与提成口径是否要在系统内体现？
8. **支付接入细节**：Visa/万事达具体走哪家收单/支付服务商？本地三方支付名称与接口资料是否已拿到？
9. **数据与合规**：是否有“数据必须在本地/私有云”的要求？AI若上，是否必须私有化？
