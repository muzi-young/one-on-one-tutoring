# 报价方案 HTML 生成技能

## 概述

基于固定的视觉模板和组件体系，为软件项目生成专业的报价方案HTML文件。输出为单个自包含HTML文件（内联CSS，无外部依赖），可直接浏览器打开。采用暖棕金色系设计，包含固定顶部导航栏（"铁肩担荣耀，妙手筑梦想"品牌标语）、双栏Hero封面、架构展示卡片、可视化时间线排期等高端视觉组件。

## 固定信息

- 公司名称：**郑州八角信息技术有限公司**（所有文档中固定使用，不可更改）
- 品牌标语：**铁肩担荣耀，妙手筑梦想**（顶部导航栏固定显示，不可更改）
- 商务名称和联系电话：每次报价前向用户确认（见第二步）

## 内容语气规则（极其重要）

生成的HTML报价方案是**直接发给客户的正式商务文档**，必须遵守以下语气规则：

1. **禁止出现任何对话式、引导性内容**：不能有"如果您需要…"、"您可以选择…"、"建议您…"等与客户对话的语气
2. **禁止AI味道的表述**：不能有"本方案旨在…"、"我们将为您…"、"以下是我们的建议…"等套话
3. **使用陈述式、专业文档语气**：直接写结论和事实，例如"系统采用前后端分离架构"而非"我们建议采用前后端分离架构"
4. **所有描述面向决策者**：假设读者是甲方负责人或技术对接人，文字简洁、准确、有判断力
5. **避免过度修饰和空话**：不要堆砌"高效"、"优质"、"完善"等无实际含义的形容词

## 核心工作流（必须严格按顺序执行）

**禁止跳过任何步骤，禁止在用户确认前生成文件。**

### 第一步：解读文档

收到用户提供的需求文档/项目资料后，先完整阅读并输出一份**需求解读摘要**，格式如下：

```
需求解读摘要
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
项目名称：xxx
项目类型：APP / 小程序 / PC系统 / 混合
覆盖端侧：用户端（安卓/iOS/H5/小程序）+ 管理后台（PC Web）
核心业务：（3~5句话概括项目核心做什么）
目标用户：xxx
核心功能模块（初步识别）：
  1. xxx
  2. xxx
  3. ...
初步技术选型判断：后端 xxx / 前端 xxx
特殊注意点：（如硬件对接、三方API集成、特殊合规要求等）
```

### 第二步：提出疑问 + 收集必要商务信息

解读完成后，**必须先收集以下三项商务信息**（每次报价都要问，不可跳过）：

1. **商务名称**：报价方案中显示的产品经理/商务对接人姓名
2. **联系电话**：报价方案中显示的联系电话
3. **意向价格和工期**：是否已有意向价格和工期目标？如果用户给出了具体数字，后续费用评估和排期按用户给出的数字倒推分配，不再按标准公式独立核算

然后列出其他需要用户澄清的问题，涵盖但不限于：

1. **功能边界**：文档中模糊或缺失的功能描述，需要明确的业务规则
2. **端侧确认**：具体需要哪些端（安卓/iOS/小程序/H5/PC后台），是否原生开发
3. **技术约束**：是否有指定技术栈、是否需要对接硬件/三方系统、是否有特殊性能要求
4. **设计要求**：是否需要UI设计、是否有品牌视觉规范、是否有参考竞品
5. **交付范围**：一期/二期如何划分、哪些功能可以延后

使用 AskQuestion 工具或直接文字提问均可，根据问题复杂度决定。

如果文档已经足够清晰、没有需要澄清的技术/功能问题，也要明确告知用户"文档描述清晰，无需额外确认"——但三项商务信息仍然必须收集。

### 第三步：等待用户回复

根据用户的回复补充和修正需求理解。如果仍有不清晰的点，继续追问，直到所有关键信息明确。

**关于意向价格和工期的处理规则：**
- 用户给出了意向总价 → 按该总价倒推各模块人天分配，确保各项合计等于意向总价
- 用户给出了意向工期 → 按该工期填写排期，Phase阶段按比例分配
- 用户说"没有"或不指定 → 按标准计费规则独立核算

### 第四步：输出方案概要并请求确认

所有问题澄清后，输出一份**方案概要预览**供用户确认，格式如下：

```
方案概要预览
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
项目名称：xxx
商务名称：xxx | 联系电话：xxx
技术选型：后端 xxx / 前端 xxx / 后台 xxx
功能模块数：约 xx 个模块
预估页面数：用户端约 xx 页 / 后台约 xx 页
预估开发工期：约 xx 个工作日
预估总价：¥xx,xxx

是否确认生成报价方案HTML文件？
```

**必须等用户明确回复"确认/可以/生成/好的/没问题"等肯定回复后，才可进入第五步。**

### 第五步：生成报价方案文件

用户确认后，按以下流程生成：

1. 读取 [template.html](template.html) 获取完整HTML骨架和CSS样式
2. 根据确认后的需求内容填充九大板块
3. 公司名称固定写"郑州八角信息技术有限公司"，顶部栏标语固定写"铁肩担荣耀，妙手筑梦想"，商务名称和联系电话填用户提供的信息
4. 计算费用数据，确保所有金额一致
5. 以项目名命名HTML文件，存放到以项目名命名的新文件夹中
6. 输出文件后，简要说明文件位置和打开方式

## 文件结构（九大板块）

生成的报价方案HTML必须包含以下九大板块，按顺序排列：

| 序号 | 板块 | HTML锚点 | 内容要求 |
|------|------|----------|----------|
| 01 | 产品概述 | #s1 | 项目定位、统计卡片（模块数/节点数/工期/总价）、建设目标 |
| 02 | 需求分析 | #s2 | 需求背景/诉求/用户画像/交付范围/假设 + 非功能性需求表格 |
| 03 | 功能清单 | #s3 | 按模块卡片组织，每个模块含功能子组+功能点列表 |
| 04 | 技术选型 | #s4 | 后端/前端/UI选型表格 + 三方服务集成表格 + 费用说明 |
| 05 | 技术难点与方案 | #s5 | 难点-方案对照表格 |
| 06 | 技术实施方案 | #s6 | 架构展示卡片（表现层/服务层/数据层/交付层）+ 方案边界说明 |
| 07 | 实施排期 | #s7 | 可视化时间线（Phase 1~4，含阶段标题、工期胶囊、任务列表） |
| 08 | 项目报价清单 | #s8 | 详细费用表格（按UI/后端/前端/后台/测试分组）+ 费用说明 + 汇总数据卡片 |
| 09 | 商务条款说明 | #s9 | 付款方式/知识产权/质保售后/交付清单 |

## 页面骨架

```
hero-topbar（固定顶部栏：标语 + 商务信息）
  → hero（双栏封面：左侧项目信息 + 右侧元数据卡片）
  → toc（目录导航）
  → s1~s9（九大板块）
  → footer（页脚）
```

### 顶部导航栏 `.hero-topbar`

- 固定定位，毛玻璃效果（`backdrop-filter:blur(18px)`），半透明深色背景
- 左侧：品牌标语"铁肩担荣耀，妙手筑梦想"，Noto Serif SC衬线体，金色（`--gold-soft`）
- 右侧：三个胶囊标签（商务姓名、报价单位、联系电话），圆角999px

### Hero 封面区（双栏布局）

- 全屏深色渐变背景，带两个半透明金色光晕装饰圆
- **左栏**（`.hero-copy`）：项目方案标签（`hero-kicker`）、项目名称（渐变文字 白→暖白→金）、副标题关键词、项目简介
- **右栏**（`.hero-side`）：方案简述卡片 + 2×N元数据卡片网格（文档类型/版本/日期/交付形态/报价单位/联系电话）
- 1024px以下自动变为单栏

### 目录导航

- 白底卡片网格，每个目录项可点击跳转到对应板块
- 序号方块使用棕金渐变背景

### 内容板块

每个板块统一结构：
```html
<section class="sec [sec-alt]" id="sN">
  <div class="container">
    <div class="sec-num">Part 0N</div>
    <h2 class="sec-title">板块标题</h2>
    <p class="sec-desc">板块描述</p>
    <!-- 具体内容 -->
  </div>
</section>
```
- 奇数板块白底（`.sec`），偶数板块暖浅灰底（`.sec .sec-alt`）

## 核心组件

### 1. 数据表格 `.tech-table`
用于需求分析、技术选型、技术难点等信息表格。深色表头（`--secondary`白字），白底行，hover暖棕透明底。

### 2. 模块卡片 `.m-card`
用于功能清单。结构：`.m-head`（渐变图标+标题+标签）→ `.m-body`（`.fg`功能子组 → `.fl`功能列表）。图标使用渐变色，按序号交替配色。三种标签类型：`.tag-core`棕金、`.tag-flow`青蓝、`.tag-fin`青绿。

### 3. 架构展示卡片 `.arch`
用于技术实施方案。深色渐变背景，按"表现层/服务层/数据层/交付层"分行展示技术栈标签（`.arch-tag`胶囊），视觉层次高于普通表格。

### 4. 报价表格 `.qt`
用于报价清单。深色渐变表头，支持行分组：
- `.gh` — 分组标题行（暖浅灰底+左侧4px棕条）
- `.st` — 小计行（暖浅底+深棕加粗）
- `.gt2` — 总计行（深色渐变底+白字）

### 5. 信息块 `.nb`
用于文字说明段落，白底圆角卡片。内含 `.nl` 编号列表（CSS counter自动编号，浅灰底棕色数字徽标）。

### 6. 统计卡片 `.sg` > `.sc`
用于汇总数据展示（总价/工期/模块数/人数/质保期等），网格布局，衬线体大数字。

### 7. 时间线排期 `.tl`
用于实施排期。左侧竖线（棕→青绿渐变），每个阶段（`.tl-item`）包含：阶段标签（`.tl-phase`）、阶段标题（`.tl-title`）、工期胶囊（`.tl-dur`）、带"→"前缀的任务列表（`.tl-tasks`）。

详细的 CSS 变量、组件类名与样式定义见 [style-guide.md](style-guide.md)。

## 文件生成规则（第五步执行时参考）

1. **读取模板**：先读取 [template.html](template.html) 获取完整HTML骨架和CSS样式
2. **替换内容**：根据确认后的需求内容填充九大板块
3. **保持样式不变**：CSS部分（`<style>`标签内）直接复用，不做修改
4. **调整数据**：Hero区元数据、报价表格数据、统计卡片数据按实际项目填写
5. **数据校验**：确保 Hero 区和报价表格总计行金额一致；商务条款分期金额 = 总价 × 比例
6. **输出文件**：以项目名命名，存放到以项目名命名的新文件夹中

### 工作量参考

- 后端：1人/天 = 6~8个接口（复杂接口1个/天）
- 前端：1人/天 = 3~8个页面（简单8个，复杂<3个）

1. **技术选型表格（PART 04）**中新增"三方服务集成"行，列出项目用到的具体服务和推荐厂商
2. **费用说明**中注明：三方服务的按量/套餐实付费用不含在开发报价内，由甲方直接向供应商支付
3. **报价表格**中的开发工时包含三方API的接口对接开发工时，但不含供应商费用
4. 根据项目需求从上表中筛选相关服务，不要把全部服务都列上去

## 注意事项

- 所有可量化/对比类内容必须用**表格**呈现
- 功能清单拆分到"按钮级"操作，无模糊表述
- 报价表格必须包含：序号、功能模块、工作内容说明、人天、单价、报价六列
- Hero区元数据与报价表格总计行必须一致
- 商务条款的分期金额需基于总价自动计算（50%/30%/20%）
- HTML文件必须自包含，所有CSS内联，除Google Fonts外无外部依赖
- 顶部导航栏标语"铁肩担荣耀，妙手筑梦想"固定不变
- 模块卡片图标使用渐变色，按序号交替配色（见style-guide.md中的配色列表）
- 实施排期必须使用时间线组件（`.tl`），不得使用简单列表
- 技术实施方案必须使用架构展示卡片（`.arch`），按层级展示技术栈
