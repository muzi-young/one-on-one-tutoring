# PRD Writer

## Purpose

Create review-ready product requirements documents from rough user ideas, client materials, meeting notes, transcripts, screenshots, mind maps, or existing drafts. Keep every requirement traceable, mark assumptions clearly, and produce a document that product, design, engineering, QA, and business stakeholders can review.

## When to Use

- 用户说：写 PRD、生成 PRD、产品需求文档、需求规格说明、功能需求说明、需求文档、需求整理、需求拆解。
- 用户提供客户聊天记录、会议纪要、访谈内容、录屏文字稿、截图说明、脑图内容、竞品描述，并希望整理成产品需求。
- 用户要求把模糊想法变成可评审、可开发、可验收的产品方案。
- 用户要求完善已有 PRD，补充流程、功能、字段、权限、异常、验收标准或待确认问题。

## Core Principles

1. **需求可追溯**：每个关键功能点必须来源于用户原始材料、用户明确指令、合理行业常识，或标注为“待确认”。
2. **事实与推断分离**：不得把推断写成事实；所有不确定内容统一放入“待确认问题”或在条目中标注“假设”。
3. **业务语言优先**：使用产品、业务、用户能理解的语言；避免数据库字段类型、接口路径、代码术语。
4. **闭环表达**：功能描述需要覆盖用户操作、系统响应、数据变化、后台处理、异常情况和验收方式。
5. **颗粒度适中**：核心功能拆到可开发、可测试的层级；按钮、筛选、提交、审核、导出、通知等关键操作不能遗漏。

## Workflow

1. **识别输入来源**
   - 明确材料类型：口头描述、客户文档、会议纪要、录屏文字稿、截图、脑图、已有 PRD。
   - 如果信息不足，先列出“已知信息”和“待确认问题”；不要凭空补齐。

2. **提炼需求上下文**
   - 梳理项目背景、目标用户、业务目标、使用场景、核心价值、成功指标。
   - 明确端侧范围：用户端、管理后台、移动端、PC 端、小程序、App、H5 等。

3. **拆解业务流程**
   - 按主流程、分支流程、异常流程整理。
   - 对关键节点说明触发条件、参与角色、系统动作、数据状态变化。

4. **编写功能需求**
   - 按“端侧 > 模块 > 子模块 > 功能点”组织。
   - 每条功能需求写清：角色、动作、对象、规则、系统反馈、异常处理、优先级、验收标准。

5. **补充数据、权限与非功能**
   - 数据：核心对象、关键字段含义、状态流转、导入导出、统计口径。
   - 权限：角色、可见范围、可操作范围、审批关系。
   - 非功能：性能、安全、兼容性、日志、审计、可用性、合规、可维护性。

6. **输出待确认问题**
   - 问题必须具体，并说明影响的功能范围。
   - 给出建议默认假设，但明确这是建议，不是事实。

## Writing Rules

- 功能需求编号使用 `FR-001`、`FR-002` 连续编号。
- 优先级默认使用 `P0/P1/P2`：P0 必须上线，P1 重要但可排期，P2 可后续优化。
- 验收标准使用可验证语句，例如“当...时，系统应...”，避免“体验良好”“操作方便”等不可验证描述。
- 信息缺失时写“待确认”，不要写虚构结论。
- 如果用户只要求轻量 PRD，可保留核心章节：背景与目标、范围、角色、业务流程、功能需求、待确认问题。

## Quality Checklist

- [ ] PRD 覆盖背景、目标、范围、角色、流程、功能、数据、权限、非功能、验收、待确认问题。
- [ ] 关键功能点可追溯到原始材料或明确标注为假设。
- [ ] 功能描述包含用户操作、系统响应、业务规则、异常处理。
- [ ] 表格字段清晰，编号连续，优先级一致。
- [ ] 所有待确认问题都说明了影响范围。
- [ ] 没有把技术实现细节误写成产品需求，除非用户明确要求。
