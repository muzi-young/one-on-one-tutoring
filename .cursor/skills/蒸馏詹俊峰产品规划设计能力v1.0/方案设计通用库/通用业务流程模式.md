# 通用业务流程模式

本文件定义了跨行业的高频业务流程模板。当项目无法匹配特定行业子场景时，AI 从此库提取可复用的流程模式，结合用户需求进行定制化组合。

---

## L1 领域核心模型

### 流程设计原则

- **闭环原则**：每个流程必须有明确的起点和终点，中间步骤形成完整闭环。
- **异常处理原则**：每个关键节点必须定义正常路径和异常路径。
- **角色明确原则**：每个步骤必须明确执行角色。

### 流程中的核心状态

- **触发**：什么事件或操作启动了流程。
- **进行中**：流程正在执行，可能有多条分支路径。
- **等待**：流程暂时挂起，等待外部输入或条件满足。
- **完成**：流程正常结束。
- **终止**：流程异常结束，需要人工介入。

## L2 动态形态参考

*请注意：以下为常见业务流程模板，AI 可根据需求自由组合、修改步骤顺序、或颠覆流程设计。*

### 流程一：用户注册与认证

1. 用户输入基本信息（手机号/邮箱）。
2. 系统发送验证码。
3. 用户输入验证码，完成基础注册。
4. [可选] 用户补充个人资料。
5. [可选] 用户提交实名认证材料。
6. [可选] 系统审核认证材料，通过后标记为已认证。
7. 注册完成，用户获得基础权限。

**核心分支**：
- 验证码发送失败 → 提示用户重试或更换联系方式。
- 实名认证不通过 → 通知用户原因，允许重新提交。
- 用户已存在 → 引导至登录流程。

---

### 流程二：交易下单与履约

1. 买方选择商品/服务，加入购物车或直接下单。
2. 买方确认订单信息（数量、规格、地址等）。
3. 买方完成支付。
4. 系统通知卖方有新订单。
5. 卖方确认订单，准备发货/提供服务。
6. 卖方发货/开始服务，系统更新订单状态。
7. 买方确认收货/服务完成。
8. 系统结算款项给卖方。
9. 双方互相评价。

**核心分支**：
- 支付失败 → 提示用户更换支付方式或重试。
- 卖方超时未确认 → 系统自动取消订单，退款给买方。
- 物流异常/服务中断 → 触发售后流程。
- 买方超时未确认 → 系统自动确认收货。

---

### 流程三：内容审核与发布

1. 创作者提交内容。
2. 系统进行自动审核（敏感词、图片识别等）。
3. 自动审核通过 → 内容直接发布。
4. 自动审核不通过 → 转入人工审核队列。
5. 人工审核通过 → 内容发布。
6. 人工审核不通过 → 通知创作者原因，允许修改后重新提交。

**核心分支**：
- 内容命中高敏感规则 → 直接拒绝，标记创作者账号。
- 特殊时期/特殊品类 → 全部转入人工审核。

---

### 流程四：投诉与纠纷处理

1. 用户发起投诉，选择投诉对象和投诉类型。
2. 系统记录投诉，生成工单。
3. 平台客服介入，查看双方提供的证据。
4. 客服尝试调解，双方协商解决方案。
5. 调解成功 → 执行解决方案，关闭工单。
6. 调解失败 → 升级为纠纷，由平台仲裁。
7. 平台给出仲裁结果，强制执行。
8. 双方可对仲裁结果进行申诉。

**核心分支**：
- 投诉证据不足 → 通知投诉方补充证据，超时未补则关闭。
- 被投诉方不回应 → 超时后平台默认采纳投诉方意见。

---

### 流程五：佣金结算与分账

1. 交易完成，系统根据预设规则计算各方佣金。
2. 生成结算单，各方可查看。
3. 各方对结算单有异议 → 发起结算申诉。
4. 各方确认结算单 → 系统执行分账。
5. 款项到达各方账户。
6. 生成结算记录，支持对账和导出。

**核心分支**：
- 结算申诉 → 冻结该笔结算，等待人工处理。
- 分账失败 → 通知相关方，重新尝试或人工处理。
