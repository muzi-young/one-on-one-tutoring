# K5 - 项目管理

## L1 领域核心模型

### 核心角色
- **项目成员**：执行具体任务，更新任务状态，提交工作成果。
- **项目经理**：创建和分配任务，监督进度，协调资源，汇报项目状态。
- **产品经理/客户**：定义需求，确定优先级，验收交付成果。
- **平台运营方**：提供项目管理的多种视图和自动化能力。

### 核心资源
- **任务与子任务**：具有明确负责人、截止日期、状态和依赖关系的工作单元。
- **项目视图**：列表、看板、甘特图、日历等多种可视化方式。
- **工作流规则**：任务在不同阶段的流转规则、触发器、自动操作。

### 核心诉求
- **项目成员**：明确知道今天要做什么，不被无关信息干扰，方便汇报进度。
- **项目经理**：实时监控项目健康度，发现阻塞和延期风险，生成进度报告。
- **客户/产品方**：了解整体进度，确保需求被正确理解并按计划交付。
- **平台**：通过团队订阅和高级功能变现，成为企业核心工作流的一部分。

### 关键矛盾
- **规划与变化**：项目需要详细规划，但实际中需求频繁变动，详细计划很快过时。
- **宏观控制与微观管理**：管理者希望看到细节，但成员感觉被微管理而不适。
- **工具灵活性与团队规范**：工具可自定义任何流程，但缺乏规范会导致每个项目的使用方式不一致。
- **个人任务与项目全局**：成员只关心自己的任务，容易忽视跨任务依赖和整体里程碑。

## L2 动态形态参考

*请注意：以下为常见业务形态，AI 可根据需求自由组合、修改或颠覆。*

### 形态一：通用看板工具
- **场景描述**：以卡片和列表为主，支持拖拽，适合敏捷团队。
- **关键特性**：上手简单，适合轻量级团队，重度依赖插件和集成。

### 形态二：专业项目管理套件
- **场景描述**：包含甘特图、关键路径、资源负荷、工时管理等全套功能。
- **关键特性**：适合大型复杂项目，学习曲线陡峭。

### 形态三：垂直场景工具
- **场景描述**：只针对软件开发（需求/缺陷管理）、建筑或影视制作等单一行业。
- **关键特性**：与该行业的工具链和术语深度融合。

### 形态四：任务+文档+知识库融合
- **场景描述**：项目管理与文档、知识库无缝连接，需求直接关联到执行任务和任务产生的文档。
- **关键特性**：减少工具切换，信息可追溯性强。
