# G1 - 天气日历

## L1 领域核心模型

### 核心角色
- **普通用户**：查看天气和日历信息的个人，用于出行决策和生活安排。
- **内容提供方**：气象数据服务商，提供天气、空气质量等数据。
- **平台运营方**：将天气和日历整合为便捷工具的开发者。

### 核心资源
- **气象数据**：实时天气、逐小时预报、空气质量、灾害预警。
- **日历数据**：公历、农历、节假日、纪念日。
- **位置服务**：精准定位当前城市和区县。

### 核心诉求
- **用户**：准确获取天气信息，合理安排出行和活动，不忘记重要日子。
- **平台**：建立用户每日启动习惯，通过广告、生活服务和增值功能变现。

### 关键矛盾
- **准确性与用户预期**：天气预报非 100% 准确，用户期望绝对精准会产生落差。
- **简洁与功能堆砌**：天气工具应保持简洁，但商业化需求会往里面堆内容如资讯、小视频。
- **隐私与个性化**：基于位置的精准预报需要用户位置信息，用户担心隐私。
- **每日固定流量与变现**：用户只看一眼就走，停留时间短，变现能力受限。

## L2 动态形态参考

*请注意：以下为常见业务形态，AI 可根据需求自由组合、修改或颠覆。*

### 形态一：纯天气工具模式
- **场景描述**：仅展示实时天气、未来预报和空气质量。
- **关键特性**：界面简洁，用户认知清晰，变现困难，适合作生态中的工具模块。

### 形态二：天气+生活指数模式
- **场景描述**：融入穿衣推荐、运动适宜度、洗车指数等生活决策。
- **关键特性**：与生活场景结合紧密，用户决策依赖度提升。

### 形态三：天气+日历融合模式
- **场景描述**：天气数据与日历日程结合，显示事件当天的天气情况。
- **关键特性**：创造独特体验，用户粘性增强，适合个人效率工具。

### 形态四：灾害预警+应急服务模式
- **场景描述**：台风、暴雨等极端天气预警功能加强，接入应急救灾信息。
- **关键特性**：公益属性强，政府和公共机构可合作提供防灾支持。
